## 4. 設計判断：単一プロジェクト vs 複数プロジェクト

### 4.1 選択肢

| 方式 | 概要 | 月額コスト |
|---|---|---|
| **A. 単一プロジェクト + instance_id 分離** ⭐ **推奨** | 1つのSupabaseプロジェクトで全店舗管理、`instance_id`で論理分離 | $25（ほぼ固定） |
| B. インスタンスごとにプロジェクト分割 | 店舗数分だけSupabaseプロジェクトを作る | $25 × 店舗数 |

### 4.2 単一プロジェクト方式（方式A）を推奨する理由

設計書 Section 4 でも既に採択済み。以下の理由から：

1. **コストが線形に増えない** — 5店舗でも$25で済む（方式Bだと$125）
2. **マスター管理画面がシンプル** — 1つのDBを操作するだけ
3. **クロスインスタンスの集計が容易** — 法人単位税務合算（Phase F）が楽
4. **開発・デバッグが容易** — 複数プロジェクトを行き来しなくていい

### 4.3 方式Aのリスクと対応

| リスク | 対応 |
|---|---|
| 1つのDB障害で全店舗停止 | 日次バックアップ＋PITR検討 |
| インスタンス間のデータ漏洩 | middlewareでinstance_id強制、RLS検討 |
| スケーリング限界 | 500店舗規模になったらEnterprise移行 |

## 5. 移行タスクとスケジュール

### 5.1 即実施できること（Phase 0 内、2026/5月まで）

1. Supabase Dashboard (https://supabase.com/dashboard/project/cbewozzdyjqmhzkxsjqo) にログイン

3. **Upgrade to Pro** ボタン押下

5. 月額/年額を選択（**年額推奨、約17%割引**）
6. プランの確認・確定

#### Step 2: プロジェクト設定の見直し

1. **Compute Instance の選定**
   - 現状: Micro（デフォルト、$10 クレジットで込み）
   - 6/1本番運用までは Micro で十分
   - リゼクシー追加時に Small（$15/月）へアップ検討

2. **バックアップ設定**
   - 日次バックアップ（7日保持）は自動で有効
   - **Point-in-Time Recovery (PITR)** は必要時に追加（+$100/月）
     - 6/1本番運用直後は不要
     - リゼクシー稼働後から検討

3. **ネットワーク設定**
   - IPv4 アドオン（$4/月）は**不要**（Vercel/Next.js からはIPv6で接続可能）

## 6. 費用対効果の分析

### 6.1 年間コスト試算

| シナリオ | 月額 | 年額 |
|---|---|---|
| Pro基本（月額契約）| $25 | $300 |
| Pro基本（年額契約）| $20.83 | **$249.96** |
| Pro + 小規模スケーリング | $30〜40 | $360〜480 |
| Pro + リゼクシー追加後 | $35〜50 | $420〜600 |

### 6.2 代替案との比較

**セルフホスト（VPS）の場合**
- Hetzner CX22 等: 月$4〜6
- ただし、運用工数が増大（バックアップ、スケーリング、セキュリティ）
- **非推奨**（T-MANAGE はビジネスアプリであり、インフラ運用に時間を取られるべきではない）

**Firebase 移行の場合**
- 同等機能で $85/月〜（Supabaseの3倍以上）
- さらに PostgreSQL→Firestore 再実装が必要
- **完全に非推奨**

### 6.3 T-MANAGE ビジネスにおけるコスト意味

- 月$25 ≒ セラピスト1名の1日分の報酬程度
- 100名規模のサロン運用基盤として破格
- 将来5店舗展開でも月$50程度で収まる見込み
