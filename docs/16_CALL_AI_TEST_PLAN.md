# 16. 通話AI 実電話テスト手順書

> **目的**: 実装済みの通話AI機能（Whisper + Claude 分析 + CTI連動）を、実際の電話で段階的にテストし、6月1日の本番稼働に向けて安定運用できる状態に持っていく。
>
> **このドキュメントの使い方**: 5/6 からテスト開始。アンジュスパさんが自身の PC で1台テスト → 問題を潰してから店舗展開。

**作成日**: 2026-04-25
**テスト期間**: 2026-05-06 〜 2026-06-15（約6週間）
**本番稼働開始**: 2026-06-01（テストは本番稼働後も継続）
**担当**: アンジュスパさん主導、必要に応じてスタッフサポート

---

## 🎯 テストの全体設計

### なぜ「マスター機1台」から始めるのか

3店舗（名古屋・安城・豊橋）に同時展開する前に、**アンジュスパさんの PC 1台**で通しテストする理由：

1. **問題発生時に原因特定しやすい**（環境変数が1つしかない）
2. **スタッフへの影響ゼロ**（失敗してもスタッフ業務に支障なし）
3. **経営責任者のアンジュスパさんが直接操作**するので、仕様理解と実装のずれが見つけやすい
4. **早期にコスト感を把握**できる（Whisper + Claude の1日の課金額）

### 段階的な導入フロー

```
【第1段階】5/6 〜 5/12 (1週間)
アンジュスパさん PC 単独テスト
  ↓ 機能単体の動作確認
【第2段階】5/13 〜 5/19 (1週間)
Android CTI 連携テスト
  ↓ 着信検知と通話録音の連動
【第3段階】5/20 〜 5/26 (1週間)
安城店で実運用シミュレーション
  ↓ スタッフ1名にテスト参加してもらう
【第4段階】5/27 〜 5/31 (5日間)
全3店舗で限定運用
  ↓ 機能ON/OFF切替可能な状態で本番同等
【本番稼働】6/1
本番運用開始（ただしテストは継続）
【第5段階】6/1 〜 6/15 (2週間)
運用データ収集・チューニング
```

## 🔬 第1段階: マスター機 PC 単独テスト（5/6 〜 5/12）

アンジュスパさんの PC で、**通話AI の各機能が単体で動くか**を確認する。この段階では Android CTI は使わない。

### テスト A-1: ブラウザから手動録音

**目的**: マイク入力 → Whisper 文字起こしの動作確認

**手順**:
1. Chrome で `/call-test` にアクセス
2. ログイン（アンジュスパさんの PIN）
3. 「録音開始」ボタンをクリック
4. マイクに向かって **5 秒ほどテスト発話**:
   ```
   「もしもし、来週の月曜日、14時で予約お願いします」
   ```
5. 「録音停止」
6. 文字起こし結果が画面に表示されるのを確認

**合格基準**:
- [ ] 文字起こしが30秒以内に完了
- [ ] 日本語が正確に起こされている
- [ ] 「予約」「月曜日」「14時」等のキーワードが拾えている

**失敗時のチェック**:
- Chrome の URL バー左にマイクアイコンが出ているか
- 開発者ツール(F12) → Console にエラーが出ていないか
- `OPENAI_API_KEY` が Vercel 環境変数に設定されているか

### テスト A-2: AI分析の動作確認

**目的**: 文字起こしテキストが Claude で正しく構造化されるか

**手順**:
1. テスト A-1 を実施後、文字起こし結果を確認
2. 「AI 分析を実行」ボタンをクリック
3. 30秒ほど待つ
4. 以下の項目が返ってきているか確認:
   - `summary`（要約）
   - `intent`（予約/問い合わせ/苦情/キャンセルの分類）
   - `sentiment`（感情：positive/neutral/negative）
   - `extracted.customer_name`（名前抽出）
   - `extracted.date_time`（日時抽出）
   - `extracted.course`（コース抽出）

**合格基準**:
- [ ] intent が `booking` と判定される
- [ ] date_time に「月曜日 14時」的な値が入る
- [ ] summary が日本語で要約されている
- [ ] エラーが出ない

### テスト A-3: 顧客マッチング

**目的**: 電話番号から既存顧客を特定できるか

**手順**:
1. T-MANAGE でテスト用顧客を1件登録
   - 名前: 「テストユーザー」
   - 電話: `090-9999-0001`
2. `/call-test` で以下のセリフを録音:
   ```
   「090-9999-0001 の者ですが、予約お願いします」
   ```
3. AI 分析実行
4. `matched_customers` に「テストユーザー」が入るか確認

**合格基準**:
- [ ] 電話番号が正確に抽出される
- [ ] 登録済み顧客とマッチする
- [ ] マッチング情報がタイムチャートに連携できる

### テスト A-4: メンズエステ特有用語の認識

**目的**: Whisper が業界用語を正しく拾うか

**テストフレーズ**:
```
「アロマとリンパのコースで、本指名でお願いします」
「オイルマッサージのロングコース 90 分でお願いします」
```

**合格基準**:
- [ ] 「アロマ」「リンパ」「本指名」「オイル」が正しく起こされる
- [ ] 「チョップ」「リフレ」「タイ古式」が入らない（除外済みのはず）

### テスト A-5: 月次予算管理の動作確認

**目的**: 使用量が記録され、予算超過時に自動停止するか

**手順**:
1. `call_usage_logs` テーブルを確認
2. テスト A-1〜A-4 実施後、当日の行が追加されているか
3. `whisper_seconds`, `sonnet_input_tokens`, `sonnet_output_tokens` が計上されている
4. 月次累計が表示されることを確認（`/call-assistant/settings` の使用量ビュー）

**擬似的に予算超過をテスト**:
1. `call_ai_settings` の `monthly_budget_usd` を `$0.01` など極小値に一時変更
2. 通話AI 機能を実行
3. 429 エラー or 警告メッセージが出ることを確認
4. 設定を元に戻す

**合格基準**:
- [ ] 使用量が自動記録される
- [ ] 予算超過時に自動停止が機能する
- [ ] 予算超過アラートが表示される

## 📱 第2段階: Android CTI 連携テスト（5/13 〜 5/19）

既存のAndroid CTI端末を**「見直し必要」な状態**から復旧させて、着信検知を安定化する。

### Android CTI 端末の状態確認

**現状の把握**:
| 項目 | 確認事項 |
|---|---|
| 端末の型番・OS バージョン | Android 10+ が望ましい |
| CTI アプリのインストール状態 | 最新ビルドが入っているか |
| `Config.kt` の設定 | Supabase URL/Key が現行のものか |
| バッテリー最適化 | 「制限なし」になっているか |
| USB ケーブル | データ通信可能なもの（充電のみNG） |

### テスト B-1: Android CTI 再ビルド（必要な場合）

**条件**: 前回ビルドから Supabase の URL/Key が変わっていたら必須

**手順**:
1. `android-cti/README.md` の「STEP 2: プロジェクトを開く」に従う
2. `Config.kt` を現行の Supabase 情報で更新
3. Android Studio で ▶ ボタンを押してビルド＆インストール
4. 端末にインストールされることを確認

### テスト B-2: 着信検知（基本動作）

**目的**: 電話が鳴ったら `cti_calls` に記録されるか

**手順**:
1. Android CTI 端末の電源を入れる
2. CTI アプリを起動（バックグラウンドで動作）
3. **別の電話**（予備のスマホ）から Android CTI 端末に発信
4. 1〜2 回鳴らして切る
5. T-MANAGE の `/cti-monitor` を確認
6. 着信ログに「予備のスマホの番号」が表示されるか

**合格基準**:
- [ ] 5秒以内に着信ログに表示される
- [ ] 電話番号が正確に記録される
- [ ] 着信日時が正確

**失敗時のチェック**:
- Android の通知権限（電話・通話履歴）が ON か
- CTI アプリがバックグラウンド実行できる状態か
- バッテリー最適化の「制限なし」が外れていないか
- Supabase の `cti_calls` テーブルに直接 insert できるか（API 疎通確認）

### テスト B-3: 着信時のポップアップ表示

**目的**: PC の T-MANAGE で着信ポップアップが表示されるか

**手順**:
1. PC で T-MANAGE を開いたままにする（`/dashboard` 等）
2. Android CTI 端末に発信
3. PC 画面に着信ポップアップが出るか確認

**合格基準**:
- [ ] 5秒以内にポップアップ表示
- [ ] 電話番号が表示される
- [ ] 既存顧客ならその情報が表示される（名前・過去の予約）
- [ ] 新規なら「新規顧客」と表示される

### テスト B-4: 既存顧客からの着信

**目的**: 着信 → 顧客特定 → 過去の予約履歴表示

**手順**:
1. テスト用顧客（テスト A-3 で登録済み）の電話番号から発信
2. ポップアップに顧客情報が出るか確認

**合格基準**:
- [ ] 「テストユーザー」として表示
- [ ] 過去の予約履歴へのリンクが機能

### テスト B-5: ブラックリスト検知

**目的**: NG顧客からの着信で警告が出るか

**手順**:
1. 顧客を1件「ブラックリスト」として登録（`customers.is_blacklisted = true`）
2. その顧客の電話番号から発信
3. ポップアップに「⚠️ ブラックリスト顧客」表示があるか

**合格基準**:
- [ ] ブラックリスト警告が目立つ色で表示
- [ ] スタッフに分かりやすい

### テスト B-6: 24時間連続稼働テスト

**目的**: 長時間稼働で Android CTI アプリが落ちないか

**手順**:
1. Android 端末を充電器につないで放置
2. 24時間後に以下を確認:
   - CTI アプリがまだ動作しているか
   - その間の着信が全て記録されたか
   - バッテリーの挙動（充電が正常に回っているか）

**合格基準**:
- [ ] アプリが途中でクラッシュしない
- [ ] 24時間中の全着信が記録される（最低10件）
- [ ] 端末が熱くなりすぎない

## 🚨 想定エラーと対処法

### エラー1: Whisper 文字起こしが失敗する

**症状**: `/api/call-analyze` で `status: 'error'` が返る

**原因候補**:
- 録音ファイルが短すぎる（1秒以下）
- マイクが壊れている / 音量が小さすぎる
- OpenAI API のレート制限
- `OPENAI_API_KEY` が未設定 or 期限切れ

**対処**:
1. ブラウザの Console で具体的なエラーメッセージを確認
2. `OPENAI_API_KEY` の有効性を OpenAI ダッシュボードで確認
3. 別の端末のマイクで試す

### エラー2: Claude が 429 を返す（レート超過）

**症状**: AI 分析が「時間をおいて再実行」と表示

**対処**:
1. `call_ai_settings.monthly_budget_usd` を確認
2. 予算超過の場合、一時的に増額 or 翌月まで停止
3. Claude 側のレート制限なら数分待つ

### エラー3: CTI 着信が記録されない

**症状**: Android 端末に着信しても T-MANAGE に表示されない

**対処チェックリスト**:
1. Android CTI アプリが起動しているか（通知バーにアイコン）
2. 電話・通話履歴の権限が ON か
3. バッテリー最適化が「制限なし」か
4. Supabase に直接接続できるか（WiFi 切断して 4G に切替テスト）
5. `Config.kt` の URL/Key が現行のものか（半年前から変わっていないか）

### エラー4: ポップアップが表示されない

**症状**: `cti_calls` に記録はされているが PC で表示されない

**対処**:
1. PC の T-MANAGE タブが開いているか（閉じていると通知されない）
2. ブラウザがアクティブか（完全な非アクティブだと不可）
3. Realtime 通信が動作しているか（F12 → Network → WS確認）
4. `cti-popup.tsx` が正しく読み込まれているか

### エラー5: 予約下書きの精度が低い

**症状**: 日時やコース名が間違っている

**対処**:
1. 原文の文字起こしを確認（そもそも起こせていない可能性）
2. AI プロンプトの調整（`/api/call-analyze/route.ts` の system 部分）
3. サンプルを蓄積して後日プロンプト改善

## 🔄 ロールバック手順

### 軽度の問題（AI 精度が低い等）

**対処**: 機能は残したまま運用調整
- `call_ai_settings.enabled = false` で一時停止
- AI プロンプト調整後に再開

### 中度の問題（API コストが想定の2倍）

**対処**: 月予算を上方修正 or エスカレーション無効化
- `call_ai_settings.allow_opus_escalation = false`
- Sonnet のみで運用

### 重度の問題（データ不整合が発生）

**対処**: 機能を完全に停止して、手動運用に戻る
1. `/call-assistant/settings` → `enabled = false`
2. 該当期間の `call_transcripts` をバックアップ後、調査
3. 問題のあるレコードを削除 or 修正
4. 再発防止策を策定後に再開

### Android CTI だけ停止（録音機能は残す）

**対処**: 着信検知だけ止めて、手動でページを開いて録音
- Android 端末の CTI アプリを停止
- PC 側は `/call-test` で手動操作
